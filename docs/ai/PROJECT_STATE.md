## 1. 项目定位

SPR 是一个面向小店渠道业务员的开单、库存、经销商出库导入、卖进报表系统。

当前主要形态：

- 前端：手机网页优先，iPhone / 移动端体验优先。
- 后端：Supabase。
- 使用方式：业务员选择员工、门店、商品后开单；系统记录销售、库存、报表。
- 未来可能迁移到小程序或国内部署，但当前以 Web + Supabase 为主。

核心目标：

1. 快速开单。
2. 业务员车销库存管理。
3. 导入经销商出货表，自动转成业务员库存。
4. 按门店、员工、商品生成卖进数据。
5. 支持售后订单标记、售后品项剔除统计和导出。

## 3. 当前核心数据流

### 3.1 经销商出库导入到库存

```text
经销商 Excel
  -> stock_jn.html / stock_ct.html
  -> raw_dealer_outbounds
  -> Supabase trigger / sync function
  -> van_stocks
```

含义：经销商出库单导入后，通过客户编号映射到业务员工号，再通过条码匹配商品，最终增加对应业务员车销库存。

### 3.2 业务员开单到销售记录

```text
员工选择
  -> 门店选择
  -> 商品选择
  -> sales_orders
  -> sales_order_items
  -> 报表 / 门店订单总览 / 导出
```

销售开单记录进入订单主表与订单明细表。

### 3.3 售后相关数据流

售后订单或售后品项必须：

- 在订单总览显示“有售后”。
- 不参与品项叠加统计。
- 不参与卖进统计的正常销量叠加。
- 管理后台导出订单详情时剔除售后项。

详细规则见 `DATA_RULE_ENGINE.md`。

## 4. 当前核心页面 / 功能

### 4.1 开单页

主要职责：

- 选择员工。
- 选择门店。
- 搜索商品。
- 按品牌 / 规格 / 口味选择商品。
- 支持不同单位换算。
- 写入订单主表和订单明细表。
- 新增订单草稿会按 `employee_code + atom_code` 缓存在浏览器 `localStorage`，刷新后回到同一员工同一门店的新增开单页应恢复已选数量、价格、拼盒展开状态和订单日期。
- 新增订单提交成功或用户放弃返回时，应清理对应草稿，避免下次误恢复旧单。

重要 UI 习惯：

- 移动端优先。
- 禁止 iOS 双击放大。
- 加载文案统一使用：`正在加载..`
- 门店搜索文案倾向：`搜门店`
- 散价选择保留 0.05 元刻度；整价选择使用 0.10 元刻度。

### 4.2 门店页

主要职责：

- 展示门店历史订单。
- 展示订单总览。
- 有售后订单时，订单总览显示“有售后”。
- 在门店搜索框有输入时点击返回，应先清空搜索框并停留在门店页，不直接返回员工页。
- 门店历史订单卡片和订单详情页需要提供明显的“生成单据”入口。
- 生成的送货单图片必须显示订单日期，下载文件名也应带日期。
- 售后品项不叠加到正常销售品项里。
- 订单详情页商品名展示应对齐管理后台开单明细 Excel 的字段来源：规格栏使用 `products.spec`，同一规格放在同一栏内；栏内口味行使用 `products.flavor`，不要把 `name` / `product_name` 当作首选展示名。

### 4.3 卖进数据页

主要职责：

- 按员工、门店、商品统计卖进数据。
- 售后项不参与销量和品项叠加。
- 有售后的订单可保留售后提示，但不要把售后商品叠加到品项汇总。

### 4.4 管理后台

主要职责：

- 导出订单详情。
- 导入库存。
- 管理后台开单明细 Excel 导出必须保留商品条码，尤其是拼盒商品不能只按名称或规格汇总。
- 跳转不同经销商导入页。

当前导入库存应区分两个经销商入口：

- 吉能：`stock_jn.html`
- 长涛：`stock_ct.html`

### 4.5 库存导入页

主要职责：

- 读取 Excel。
- 按经销商格式映射列。
- 写入 `raw_dealer_outbounds`。
- 触发库存同步。

详细映射见 `IMPORT_MAPPING.md`。

### 4.6 当前前端代码结构

当前仍以静态 HTML + 原生 JavaScript 为主，不引入构建系统。

门店相关入口页：

- `store.html`：门店首页入口。
- `store_stock.html`：库存查看入口。
- `store_report.html`：卖进数据入口。
- `store_new.html`：新门店开单入口。

门店页共享脚本按顺序加载：

1. `store-app.js`：门店、历史订单、订单详情、开单、卖进数据、库存查看等主流程。
2. `store-delivery-note.js`：送货单 / 生成单据相关逻辑，包括订单明细聚合、送货单 HTML、图片生成和下载。
3. `store-mix-box-edit-fix.js`：拼盒编辑恢复补丁。
4. `store-qty-popup.js`：数量选择交互。
5. `store-after-sales.js`：售后显示、售后字段处理和售后覆盖逻辑。

后续优化原则：

- 优先按功能边界小步拆分，不一次性重写框架。
- 拆分时保持全局函数名兼容现有 HTML `onclick` 调用。
- 送货单逻辑应继续留在 `store-delivery-note.js`，不要再塞回 `store-app.js`。
- 涉及售后、金额、库存、导出和品项叠加时，仍以 `DATA_RULE_ENGINE.md` 为最高业务规则来源。

## 5. 当前核心数据库表概览

> 详细结构以 `SCHEMA_LOCK.md` 为准。

主要表：

- `employees`：业务员基础信息。
- `employee_store_assets`：业务员与门店关系 / 门店资产。
- `dealer_employee_mappings`：经销商客户编号到业务员工号映射。
- `products`：商品资料，条码、品牌、规格、口味、价格、包装数。
- `van_stocks`：业务员车销库存。
- `raw_dealer_outbounds`：经销商原始出库记录。
- `sales_orders`：订单主表。
- `sales_order_items`：订单明细表。

## 6. 已知重要业务规则摘要

### 6.1 售后规则

最新原则：

- 有售后的订单，在门店页订单总览显示“有售后”。
- 有售后的订单，在卖进数据里的订单总览也显示“有售后”。
- 售后的产品不参与品项叠加。
- 管理后台导出订单详情时，售后项剔除出 Excel。

### 6.2 库存规则

- 经销商出库导入增加业务员库存。
- 业务员正常销售减少业务员库存。
- 售后是否影响库存必须按 `DATA_RULE_ENGINE.md` 规则处理，不能在页面里临时写一套算法。

### 6.3 商品包装规则

- `package_reg <= 0` 时，优先使用 `products.pcs_per_case` 回填。
- 条码是商品匹配核心字段。
- Excel 条码必须按文本处理，避免科学计数法。
- 拼盒订单明细在管理后台开单明细 Excel 导出时，应按每个散装商品条码拆成独立散数行；散价 = 拼盒整价 / `products.pcs_per_box`，保留两位小数。
- 门店送货单图片中，拼盒可以按中盒数量展示；管理后台 Excel 导出为了对账和条码准确性，必须按散条码拆行。

## 7. 历史踩坑与注意事项

### 7.1 Supabase 默认行数限制

Supabase 前端查询默认可能只返回 1000 行。员工门店关系、商品、库存等大表查询必须分页或合理范围查询，不能默认只取一次。

### 7.2 CSV / Excel 中文乱码

导出给 Excel 打开时，CSV 需要 UTF-8 BOM。否则中文可能乱码。

### 7.3 Excel 条码科学计数法

条码必须按文本处理。不要让 Excel 自动把条码转成科学计数法。

### 7.4 时区问题

Supabase 常用 UTC 存储，但前端展示和业务日期应按中国时区 / 本地业务日处理。不要让 UTC 日期导致订单日期偏移。

### 7.5 外键阻断 TRUNCATE

`van_stocks` 等表可能引用 `products`。清空表时注意外键依赖，必要时按依赖顺序清空或使用 `TRUNCATE ... CASCADE`，但 CASCADE 前必须确认影响范围。

### 7.6 RLS / 权限

历史阶段中 RLS 可能关闭过。修改权限策略前必须确认当前部署方式，不要随意假设登录体系已经存在。
