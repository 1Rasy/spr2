# SPR 数据规则引擎

> 本文件是 SPR 项目的业务计算规则最高优先级文档。
>
> 任何页面、SQL、导出、报表、库存逻辑，都不得自行定义与本文件冲突的计算口径。

---

## 1. 最高原则

### 1.1 唯一规则源

所有涉及以下内容的逻辑，必须以本文件为准：

- 订单是否计入销售。
- 售后是否计入汇总。
- 商品品项是否叠加。
- 库存如何增减。
- 导出 Excel 时哪些项保留或剔除。
- 经销商出库如何转成业务员库存。

### 1.2 禁止分散规则

禁止在不同页面、不同 SQL、不同导出函数里各写一套口径。

如果发现以下情况，必须重构为统一规则：

- 门店页统计和卖进数据统计结果不一致。
- 页面显示的销量和导出的销量不一致。
- 售后品项在某个页面被剔除，在另一个页面又被叠加。
- 库存扣减和订单明细计算口径不一致。

## 2. 订单基础定义

### 2.1 订单主表

订单主表表示一次业务员对某个门店的开单行为。

通常字段包括：

- 订单 ID。
- 员工编号。
- 门店编号。
- 门店名称。
- 订单日期。
- 订单金额。
- 备注。

### 2.2 订单明细表

订单明细表示订单中的每个商品行。

通常字段包括：

- 订单 ID。
- 条码。
- 数量。
- 单价。
- 赠品数量。
- 退回 / 售后数量。
- 售后处理方式。

实际字段名以 `SCHEMA_LOCK.md` 和当前数据库为准。

---

## 3. 售后规则

### 3.1 售后订单定义

一个订单只要存在任意售后品项，即视为“有售后订单”。

售后品项判断规则：

```text
return_qty > 0
或 return_handle 有有效售后处理值
或 当前代码中明确标记为售后项的字段为真
```

如果字段命名不一致，修改代码前必须先确认当前数据库真实字段。

### 3.2 订单总览展示规则

门店页订单总览：

- 如果订单存在售后项，显示“有售后”。
- 正常商品仍可正常展示。
- 售后商品不应该和正常商品叠加成同一个正常销售品项。

卖进数据里的订单总览：

- 如果订单存在售后项，显示“有售后”。
- 售后商品不参与正常品项叠加。

### 3.3 品项叠加规则

售后商品不参与任何正常销售品项叠加。

具体表现：

- 不计入商品销量汇总。
- 不计入门店商品销售汇总。
- 不计入员工商品销售汇总。
- 不计入商品排行。
- 不计入卖进数据的正常销量。
- 不作为正常销售品项显示在订单汇总的叠加结果里。

### 3.4 导出 Excel 规则

管理后台导出订单详情时：

```text
售后项必须从导出 Excel 的订单明细中剔除。
```

也就是：

- 正常销售项：导出。
- 赠品项：按当前导出规则处理，若原系统导出赠品则保留。
- 售后项：剔除。

### 3.5 售后金额规则

当前稳定原则：

- 售后项不计入正常销售金额。
- 订单主表金额如果历史上已经包含了售后影响，不要在未确认前二次计算覆盖。
- 报表统计金额时，应优先按正常销售明细重新聚合，而不是盲信订单主表 `total_amount`。

### 3.6 售后库存规则

售后对库存的影响必须明确区分处理方式。

推荐规则：

```text
正常销售：减少 van_stocks
售后恢复 / 可再销售：回补 van_stocks
售后报废 / 不可再销售：不回补 van_stocks
```

如果当前代码没有稳定区分恢复和报废，不要擅自改库存逻辑。必须先确认字段含义。

## 4. 正常销售统计规则

### 4.1 卖进口径

卖进数据 = 正常销售订单明细的聚合。

不包含：

- 售后项。
- 已明确标记为剔除统计的异常项。

是否包含赠品，需要按当前业务需求确认；默认不要把赠品当作有金额销售。

### 4.2 聚合维度

常用维度：

- 员工编号 `employee_code`
- 门店编号 `atom_code` / `store_atom_code`
- 门店名称 `store_name`
- 商品条码 `barcode`
- 商品品牌 `brand`
- 商品规格 `spec`
- 商品口味 `flavor`
- 订单日期 `order_date`

### 4.3 品项合并键

同一商品是否合并，优先使用条码 `barcode`。

不要只用商品名称合并，因为同名商品可能规格不同。

## 5. 库存规则

### 5.1 库存表定位

`van_stocks` 表示业务员当前车销库存。

常见唯一键：

```text
employee_code + product_barcode
```

### 5.2 入库来源

经销商出库导入是业务员库存的主要入库来源。

数据流：

```text
raw_dealer_outbounds
  -> customer_code 映射 employee_code
  -> barcode 匹配 products
  -> package_reg / qty 计算数量
  -> 写入或更新 van_stocks
```

### 5.3 出库来源

业务员开单的正常销售项会减少库存。

```text
van_stocks = 原库存 + 经销商入库 - 正常销售 + 可回补售后
```

### 5.4 售后与库存

售后不等于正常销售。

- 售后项不能当作新的正常销量扣减。
- 是否回补库存，取决于售后处理方式。
- 报废类售后不应回补可销售库存。

## 6. 商品包装与数量换算规则

### 6.1 条码优先

商品匹配必须优先使用条码。

不要只用商品名称匹配。

### 6.2 package_reg 回填

经销商出库导入时：

```text
如果 Excel 中 package_reg <= 0 或为空：
    使用 products.pcs_per_case
```

### 6.3 最小单位原则

涉及库存和跨页面统计时，数量应尽量统一为最小可计算单位。

不同页面可以展示“件 / 盒 / 个”，但底层计算不能混乱。

### 6.4 拼盒商品

拼盒商品用于同一品牌 / 规格下按不同口味散数拼成中盒销售。

当前稳定规则：

- 开单时拼盒数量按散数选择，但提交前必须满足整盒数量校验。
- 拼盒库存扣减应按各条码实际散数扣减。
- `sale_unit` 标记为 `拼盒` 的订单明细，`sale_qty` 表示该条码的散数，`sale_unit_price` 表示拼盒整价。
- 管理后台开单明细 Excel 导出时，拼盒必须拆成每个商品条码的散数行，`散价 = sale_unit_price / products.pcs_per_box`，保留两位小数。
- 门店送货单可以把拼盒折算为中盒展示，但不能影响管理后台 Excel 按条码拆散导出的规则。

### 6.5 特殊商品

如果存在奥利奥 3 片装或其他特殊换算商品：

- 正常销售单位、赠品单位、退回单位可能不同。
- 必须在规则中单独列明。
- 不得只在某个页面临时硬编码。

## 7. 经销商出库导入规则

### 7.1 原始表原则

经销商 Excel 原始数据应先进入 `raw_dealer_outbounds`。

不要直接写 `van_stocks`，除非是明确的人工修正功能。

### 7.2 唯一键原则

建议用以下组合避免重复导入：

```text
order_no + barcode
```

如果同一单号同一条码可能存在多行，需要先确认业务真实情况，再调整唯一键。

### 7.3 客户编号映射

经销商客户编号通过 `dealer_employee_mappings.customer_code` 映射到业务员 `employee_code`。

没有映射的记录：

- 可以保留在 raw 表。
- 不应进入业务员库存。
- 应提供排查方式。

### 7.4 商品条码校验

条码在 `products` 中不存在时：

- 可以保留 raw 原始记录。
- 不应写入有效库存。
- 应提供未匹配商品清单。

## 8. 导出规则

### 8.1 Excel 文件格式

导出的 Excel 必须保证：

- 文件扩展名和实际格式一致。
- 中文不乱码。
- 条码按文本导出。
- 日期格式清晰。
- 管理后台开单明细导出必须保留商品条码。

### 8.2 订单详情导出

订单详情导出必须遵守售后剔除规则：

```text
导出前过滤售后项。
```

### 8.3 拼盒导出

管理后台开单明细 Excel 导出遇到 `sale_unit = 拼盒` 时：

- 不按整盒聚合到单一规格行。
- 按每个 `barcode` 拆成独立散数行。
- `散数` 使用该条码对应的 `sale_qty`。
- `散价` 使用 `sale_unit_price / products.pcs_per_box`，保留两位小数。
- `整数` / `整价` 留空。
- 金额保留订单明细原金额聚合结果。

### 8.4 门店分组显示

如果做美化导出：

- 可以按门店分组。
- 不同门店可以用白色 / 浅灰交替底色。
- 不要为了美观破坏原始数据准确性。

---

## 9. 禁止事项

禁止：

1. 未读本文档就修改统计口径。
2. 在一个页面里临时修正售后逻辑，但不改报表和导出。
3. 只改前端显示，不处理导出。
4. 只改导出，不处理页面统计。
5. 使用商品名称替代条码做库存核心匹配。
6. 让 Excel 自动处理条码数字。
7. 在未确认字段含义前修改售后库存回补。
